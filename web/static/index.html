Quick Commands
What is RQL?

RQL (Rawth Query Language) is a query language I invented in about 20 minutes. It has 8 commands and zero ambiguity.

The syntax is basically: VERB key "value". The command names are deliberately stupid - SHOVE stores, YOINK retrieves, YEET deletes.

The lexer is hand-written, byte-by-byte. The parser is recursive descent with nothing to recurse into.

Architecture

Under the Hood

Every layer built without external dependencies. Not because it's smart, but because that was the point.

01 / PAGER

Disk I/O Layer

Fixed-size 4KB pages read and written directly via ReadAt/WriteAt. Buffer pool caches hot pages. Free list recycles deleted pages. Every file starts with 4 magic bytes: RAWT.

offset = pageID × 4096
02 / B+TREE

Storage Structure

Disk-based B+Tree. Leaf nodes are linked for sequential scans. Internal nodes hold keys and child page IDs. Splits when the serialized node exceeds ~4089 bytes. No node merging on delete - too much work.

search → split → serialize → flush
03 / ENGINE

Storage API

Wraps the B+Tree with TTL support (lazy expiration - if you never read an expired key, it stays in the file forever like a ghost), thread-safe locking, and runtime stats.

Engine → B+Tree → Pager → Disk
04 / RQL

Query Language

Hand-written lexer tokenizes byte-by-byte. Recursive descent parser (there's nothing to recurse into - it's a flat switch statement with a fancy name). No regex. No parser generators.

input → tokens → AST → execute
05 / TCP

TCP Interface

Redis-style TCP server on port 6379. One goroutine per connection. Line-delimited RQL in, text responses out. Connect with nc localhost 6379 and it works. That's it.

nc localhost 6379
06 / HTTP

HTTP + WebSocket

REST API for queries. WebSocket implementation hand-rolled from RFC 6455 because the standard library doesn't include one and I wasn't going to add a dependency for 200 lines of code. Static files embedded via go:embed.

POST /api/query → ws://host/ws

The Manifesto

BY NIKHIL SINGH - APRIL 2026 - NOT A LEARNING EXERCISE, SORT OF

I didn't build this to learn. I didn't build this because I had a gap in the database market I was trying to fill. I built this because every senior engineer I've ever met talks about "understanding the fundamentals" and then immediately reaches for Prisma. I wanted to see what happened if you just... didn't.

The result is rawth. A key-value store that stores your data in a custom binary format on disk, organized by a B+Tree I wrote myself, accessible via a query language I named after internet slang. It runs on a TCP server. It also has a WebSocket interface because that's what 2024 demands of you, apparently.

"While everyone else was adding another layer of abstraction, I was writing bytes to disk with a hex editor open in one window and the B+Tree Wikipedia page open in another."

Let's be honest about what this is and isn't. It is not well-optimized. The TTL expiration is lazy - keys expire only when you touch them, which means expired data can absolutely sit on disk indefinitely if you stop asking for it. The delete operation doesn't merge underflowing nodes because that is genuinely hard to implement correctly and I had other things to do. The buffer pool eviction policy is "if the map is full, don't cache it," which is not a policy any serious database person would recommend.

The B+Tree is real. It splits, it serializes to 4096-byte pages, the leaves are linked so you can scan in order. When I first got a split working correctly at the root level - the moment a tree grows from one level to two - I felt genuinely unreasonable levels of satisfaction. This was immediately followed by three hours of debugging why the page offsets were off by one whenever I had more than 47 keys. It was an integer overflow. It is always an integer overflow.

The query language was designed in about 20 minutes, which is apparent. SHOVE, YOINK, YEET, PEEK. The parser is technically recursive descent, which is a term that sounds very impressive until you realize that in this case there's nothing to recurse into because the grammar is completely flat. The hand-written lexer does handle escape characters inside strings though, so at least there's that.

The WebSocket implementation reads RFC 6455 and implements the handshake, frame parsing, and masking entirely from scratch. This was either the most educational or most pointless thing I've done all month, depending on who you ask. The standard library does not include a WebSocket server. I was not going to add a dependency for 200 lines of code. So here we are.

Is this production-ready? Absolutely not. There's no replication, no write-ahead logging, no crash recovery, no authentication. If your process dies mid-write you might corrupt the file. The "free page recycling" via a singly-linked free list is correct in theory and largely untested under load. You should not use this to store anything you care about. You should absolutely use this to understand how the databases you use every day work under the hood.

The magic bytes at the start of every file are RAWT. If you open a rawth database in a hex editor, that's the first thing you'll see. After that: the version number (1), the page size (4096), the page count, and the root page ID. Everything else is tree nodes, packed as tight as I could get them into 4KB blocks. It is, in a small way, beautiful. In a larger way, it's just bytes. Bytes I put there on purpose, which is kind of the whole point.

FAQ

Technical Questions
You Might Have

Answers that are technically correct and slightly dismissive, as all good FAQ sections should be.

4096 bytes is the default page size for most Linux filesystems and aligns with hardware sector sizes. The OS already thinks in 4KB chunks, so aligning to that avoids partial page reads and keeps I/O amplification minimal. It's also the same size SQLite uses by default, and SQLite is more correct than I'll ever be. Could it be configurable? Yes. Is it? No. Making it configurable means testing every possible page size, which means more code, which means more bugs. The current implementation has a hard-coded constant and it works fine.
It's a deliberate omission, which is a more professional way of saying it was too much work and the project still passes all its tests without it. Node merging (also called coalescing) is required for a proper B+Tree to maintain its fill-factor invariants over time. Without it, after many deletes, you end up with a lot of under-filled pages that waste disk space. For a database serving production traffic, this is a real problem. For a demo database running on a VPS, the disk space cost is probably measured in bytes. The free list does correctly recycle completely empty pages. The half-empty ones just stay half-empty. This is fine.
Lazy expiration means a key is only checked for expiration when someone tries to access it. If you SHOVE temp "hello" TTL 10 and then never touch temp again, it will sit in the file indefinitely. The TTL metadata is stored in a Go map[string]time.Time in memory, which means it's also completely lost on process restart. This is roughly how Redis does lazy expiration too, though Redis also runs a background job to periodically scan for expired keys. I did not implement a background job. The tradeoff is simplicity versus correctness. I chose simplicity. You're welcome.
The entire database engine - pager, B+Tree, storage engine, RQL parser - has zero external dependencies. Adding one for the WebSocket layer felt intellectually inconsistent. RFC 6455 is also not that long. The handshake is an HTTP upgrade with a SHA-1 hash of the key, the frame format is a small binary header, client-to-server frames are XOR masked with a 4-byte key. The implementation is about 200 lines and handles everything the terminal demo needs. It doesn't handle ping/pong frames or large message fragmentation, but nobody's sending a 100MB message through a database query terminal.
SET/GET/DEL is Redis. This is not Redis. The names are distinctive enough that you immediately know you're talking to rawth, not accidentally connected to a real Redis instance and about to do something production-breaking. There's also a mild amusement factor that I thought made the project more entertaining. The command names are technically arbitrary identifiers - the lexer would work equally well if they were called ALPHA/BETA/GAMMA. They are not called ALPHA/BETA/GAMMA because that would be confusing and less fun.
No. There is no write-ahead log, so an unclean shutdown mid-write can corrupt the file. There is no replication. There is no authentication - anyone who can reach port 6379 or 8080 can run NUKE and delete everything. The buffer pool uses a simple map with no LRU eviction policy. None of this is a secret; it's the entire premise of the project. rawth exists to demonstrate how a database works, not to replace the one your company runs on. That is not a question, but yes, you are correct. A proper buffer pool uses LRU (least recently used) or Clock eviction to keep the most-accessed pages in memory and evict cold pages when space is needed. The current approach evicts nothing - once the buffer is at 256 pages, new pages just don't get cached. For a small dataset that fits in 256 pages (about 1MB of page data), this is fine and the cache hit rate is effectively 100%. For a large dataset, you'd see more disk reads than necessary. Implementing LRU adds about 50 lines of code and a doubly-linked list. I didn't feel like it.
